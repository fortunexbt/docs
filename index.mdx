## Read the manual in layers

<Steps>
  <Step title="Inspect the architecture">
    See the operating surface, the trust boundary, and the exact verification hooks for each system.
  </Step>
  <Step title="Run the smallest honest demo">
    Follow recipes designed to work with seeded data, local providers, or a documented install path.
  </Step>
  <Step title="Read the proof ledger">
    Learn which evidence belongs in a build, release, or public capability claim.
  </Step>
  <Step title="Enter the labs carefully">
    Barter and SecurePath are documented with implemented, simulated, and planned behavior kept separate.
  </Step>
</Steps>

## Evidence ledger

| Label | It means | It does not mean |
|---|---|---|
| **Shipped** | A release or public use path exists | Every possible deployment is supported |
| **Verified** | A reproducible check or inspectable artifact backs the claim | A badge passed once |
| **Experiment** | Real code operates inside a deliberately limited envelope | Production-ready |
| **Simulated** | The behavior is mocked or emulated on purpose | Connected to the named external system |
| **Planned** | The direction is explicit and not implemented | “Almost done” |

<Note>
  Architecture and lab claims were reviewed against the public repositories on **21 July 2026**. Product-specific documentation remains the authority for current installation and API details.
</Note>
